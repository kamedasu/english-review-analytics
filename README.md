# English Review Analytics

Notionに蓄積した英会話レビューを取得し、月単位で学習状況を可視化するStreamlitアプリです。

## 実装計画

Phase 1では、ローカルで動く最小構成を優先します。

1. Notionページ本文を取得する
2. Markdown構造からレビュー単位に分割する
3. `Review` / `PhraseCard` に構造化する
4. Streamlitで月次ダッシュボードを表示する

Phase 2では、以下を追加しやすい土台を入れています。

- `state.json` によるページ単位の差分取得
- `local_store.py` / `s3_store.py` による保存先の分離
- `rag/chunker.py` によるRAG向けチャンク化
- `llm_summary.py` による月次サマリー生成の差し替え口
- `reuse_detector.py` による復習・再利用判定ロジックの分離

## セットアップ

```bash
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env
```

Notion APIを使う場合は `.env` に設定します。

```env
NOTION_API_KEY=secret_xxx
NOTION_PAGE_IDS=page_id_1,page_id_2
NOTION_ACTIVE_PAGE_IDS=active_month_page_id
ACTIVE_MONTHS=2026-05
ARCHIVED_MONTHS=2026-04
OPENAI_API_KEY=
OPENAI_MODEL=gpt-4.1-mini
DATA_ROOT_DIR=
```

NotionページIDは、月別ページをカンマ区切りで指定します。未設定の場合、画面に設定不足のエラーを表示します。

## データ保存先

保存先ベースディレクトリは `DATA_ROOT_DIR` で指定できます。未設定の場合は従来通りプロジェクト直下の `data/` を使います。

```env
DATA_ROOT_DIR=/Users/you/Library/Mobile Documents/com~apple~CloudDocs/english-review-analytics-data
```

指定したディレクトリ配下に `raw/`, `processed/`, `state/` を作って使います。iCloud Drive や Google Drive の同期フォルダを指定すると、ローカル保存済みデータを複数環境で共有しやすくなります。

## Active / Archived 月

Notion API呼び出しを減らすため、月をactiveとarchivedに分けます。

- `archived_months`: 確定済みの月です。通常表示でもSyncでもNotion APIを見に行きません
- `active_months`: 追記される可能性がある月です。通常表示ではローカル保存済みデータだけを使い、`Sync from Notion` を押したときだけNotion APIを見に行きます

推奨設定は、同期対象のactive月ページだけを `NOTION_ACTIVE_PAGE_IDS` に入れることです。

```env
NOTION_ACTIVE_PAGE_IDS=active_month_page_id
ACTIVE_MONTHS=2026-05
ARCHIVED_MONTHS=2026-04
```

`NOTION_ACTIVE_PAGE_IDS` が未設定の場合は、既存stateから `ACTIVE_MONTHS` に一致するページIDを探します。初回やstateがない環境では特定できないため、`NOTION_ACTIVE_PAGE_IDS` の設定を推奨します。

## OpenAI API設定

月次サマリーでOpenAI APIを使う場合は `.env` に `OPENAI_API_KEY` を設定します。

```env
OPENAI_API_KEY=sk-...
```

モデルを変更したい場合は `OPENAI_MODEL` を設定します。未指定時は `.env.example` の既定値を使います。

```env
OPENAI_MODEL=gpt-4.1-mini
```

`OPENAI_API_KEY` が未設定、またはAPI呼び出しに失敗した場合、アプリはrule-based summaryに自動でフォールバックします。画面の `Monthly` / `Quarterly` / `Yearly` Summary に `Generated by OpenAI API` または `Generated by rule-based fallback` と表示されます。

## 起動

```bash
streamlit run app.py
```

通常表示ではローカル保存済みの `processed/reviews.json` と `state/state.json` だけを使い、Notion APIは呼びません。

サイドバーの `Sync from Notion` を押したときだけ、active月ページをNotion APIから取得します。同期時は月ページを取得したあとreview単位で `date + content_hash` を照合し、既存reviewはスキップして、新規または変更されたreviewだけを保存します。同期結果には追加件数、更新件数、スキップ件数、同期月が表示されます。

## 集計単位の切り替え

画面上部の `Aggregation` で集計単位を切り替えられます。

- `Monthly`: 月単位で集計し、月次サマリーを表示します
- `Quarterly`: 四半期単位で集計し、四半期サマリーを表示します。例: `2026-Q2`
- `Yearly`: 年単位で集計し、年次サマリーを表示します。例: `2026`

どの集計単位でも、以下の指標を表示します。

- 総勉強時間
- 学習日数
- 最長連続学習日数
- review数
- phrase数
- reused phrase数

LLMサマリーは `Monthly` / `Quarterly` / `Yearly` に対応しています。`OPENAI_API_KEY` が未設定の場合は、どの集計単位でもrule-based summaryにフォールバックします。

## 画面構成

通常UIは実運用向けに、以下の順で表示します。

1. Aggregation切り替え
2. KPIカード
3. Summary
4. Improvement Focus
5. Reused Phrase Candidates
6. Phrase Cards
7. Reviews

raw review本文やraw match結果は通常UIには表示しません。詳細を確認したい場合はNotion側のレビュー本文を確認します。

## Improvement Focus

新しいレビュー構造では `Weak points` と `More natural expressions` を扱えます。

`Weak Points` では、期間内レビューの弱点を重複に近い内容ごとにまとめて表示します。

`More Natural Expressions` では、以下を表形式で表示します。

- `Your phrase`
- `More natural`
- `Note`
- `Date`

古いレビューには `Weak points` や `More natural expressions` が存在しない場合があります。これは正常な状態として扱い、該当項目がない期間では「まだ記録はありません」というメッセージを表示します。

## Phrase Cards

`Phrase Cards` ではpriorityで表示対象を絞り込めます。現在は `High` / `Medium` を初期選択にしています。

`Low` など別のpriorityがNotionデータに追加された場合も、選択肢として表示されます。

表示形式は以下を切り替えられます。

- `Dedupe Phrase Cards`: 同じphraseを1行にまとめます
- `Raw Phrase Cards`: 元のphrase cardをそのまま表示します

`Dedupe Phrase Cards` では以下を確認できます。

- `first_seen_date`
- `last_seen_date`
- `occurrence_count`
- `highest_priority`

## Reused Phrase Candidates

`Reused Phrase Candidates` はphrase単位に集約して表示します。

- `first_seen_date`: phrase cardとして最初に出てきた日
- `reused_dates`: 後続レビュー内で再登場した日
- `reuse_count`: 再登場した日の数
- `matched_fields`: 再登場を検出したフィールド
- `last_used_date`: 最後に再登場した日
- `retention_label`: `New` / `Reused` / `Retained` / `Strong` の簡易定着度

詳細な元データを確認したい場合は、Notion側のレビュー本文を確認します。アプリ画面では通常運用向けに集約結果のみ表示します。

## Notion本文の前提

月別ページ本文に、以下のようなMarkdown相当の構造が複数件並んでいる前提です。

```markdown
# 2026-05-03 review
- Date: 2026-05-03
- Duration: 15
- Topic: ...
- Good points:
  - ...
- Expressions to add:
  - ...
- Expressions to use next time:
  - ...
- Comment: ...

## Phrase Cards
### Card 1
- Phrase: ...
- Meaning: ...
- Example: ...
- Next review date: ...
- Priority: ...
```

Notion APIから取得したブロックは `src/notion_client.py` でMarkdown風テキストに変換し、`src/notion_parser.py` で構造化します。

## Notion実データ確認手順

1. `.env` に `NOTION_API_KEY` と `NOTION_ACTIVE_PAGE_IDS` を設定する
2. `streamlit run app.py` で起動する
3. `Sync from Notion` を押す
4. 画面上部に警告が出ていないことを確認する

取得に失敗した場合は、サイドバーのメッセージと `data/raw/` を確認します。

- Notion APIキーが正しいか
- `NOTION_ACTIVE_PAGE_IDS` が正しいか
- Notion Integrationに対象ページの権限があるか
- `data/raw/` に保存されたMarkdownが期待する構造になっているか

Notionから取得した各月ページのMarkdownは `data/raw/` に保存されます。

## Parser失敗時に見る場所

まずサイドバーの取得メッセージを見て、対象ページが取得できているか確認します。`エラー` が出る場合はNotion APIキー、ページID、Integrationの権限を確認してください。

次に `data/raw/` で、Notion本文が期待するMarkdown構造に変換されているか確認します。特に以下が重要です。

- レビューの先頭が `# 2026-05-03 review` の形式になっているか
- `- Date: 2026-05-03` が存在するか
- `## Phrase Cards` が存在するか
- 各カードが `### Card 1` の形式になっているか
- フィールド名が `Duration`, `Topic`, `Good points`, `Expressions to add`, `Expressions to use next time`, `Comment` と一致しているか

`duration_minutes` が0、`topic` が空、`phrase_cards` が0になる場合は、`data/raw/` の該当Markdownと [src/notion_parser.py](src/notion_parser.py) の正規表現・フィールド名を見比べてください。
